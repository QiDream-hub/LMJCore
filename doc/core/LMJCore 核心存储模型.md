# LMJCore 核心存储模型

> **一句话定义**：  
> LMJCore 通过两片物理空间——**集合区（`set`）** 与 **主存储区（`main`）**——将任意嵌套的数据结构拆解为扁平的、可高效访问的片段，并通过全局唯一指针重新组合。

---

## 1. 两片存储空间

LMJCore 的全部数据存储在两个 LMDB 数据库中：

| 空间        | 名称     | 用途                                      | 关键特性                                                                                      |
| --------- | ------ | --------------------------------------- | ----------------------------------------------------------------------------------------- |
| **集合区**  | `set`  | 存储**实体的关联项集合**  <br>（如对象的成员名列表、集合的元素列表） | - 启用 `MDB_DUPSORT`  <br>- 每个 Key 可关联多个 Value  <br>- **Value 按字典序自动排序**  <br>- **不保留插入顺序** |
| **主存储区** | `main` | 存储**具体值**  <br>（如对象成员的实际内容）             | - 标准 KV 结构  <br>- Key = `[17B 实体指针][成员名]`  <br>- 成员天然隔离、点查 O(1)                           |

> 💡 所有实体（对象或集合）均由一个 **17 字节全局唯一指针**标识（首字节：`0x01`=对象，`0x02`=集合），该指针即为它们在 `set` 和 `main` 中的“身份证”。

---

## 2. 对象如何存储？

以对象 `{ name: "Alice", age: 30 }` 为例：

### 步骤

1. 生成对象指针 `ptr_obj = 01...`（17 字节）
2. 在 `set` 中注册成员名（顺序无关）：

   ```text
   set[ptr_obj] → "age"
   set[ptr_obj] → "name"
   ```

→ LMDB 内部会按 `"age" < "name"` 排序存储，但这**不影响语义**
1. 在 `main` 中存储具体值：

   ```text
   main[ptr_obj + "name"] → "Alice"
   main[ptr_obj + "age"]  → 30（二进制编码）
   ```


### 存储结构图示

```
set（集合区）：
┌──────────────┬─────────────┐
│ Key: ptr_obj │ Value: "age"  │ ← 成员名集合（无序）
├──────────────┼─────────────┤
│ Key: ptr_obj │ Value: "name" │
└──────────────┴─────────────┘

main（主存储区）：
┌───────────────────────┬─────────────┐
│ Key: ptr_obj+"name"   │ Value: "Alice" │ ← 独立格子，由“指针+名称”定位
├───────────────────────┼─────────────┤
│ Key: ptr_obj+"age"    │ Value: 30      │
└───────────────────────┴─────────────┘
```

> ✅ 对象成员的**存在性由 `set` 定义**，**值由 `main` 提供**。两者必须一致，否则视为数据损坏。

## 3. 集合如何存储？

以集合 `["banana", "apple"]` 为例：

### 步骤

1. 生成集合指针 `ptr_set = 02...`
2. 在 `set` 中写入元素：

   ```text
   set[ptr_set] → "banana"
   set[ptr_set] → "apple"
   ```

3. **LMDB 自动按 Value 字典序排序**，实际存储为：
   ```text
   set[ptr_set] → "apple"
   set[ptr_set] → "banana"
   ```


> ⚠️ **重要**：`set` **不保证插入顺序**。集合是无序的，元素自动去重并按字典序排序。

### 3.1 集合的核心特性

|特性|说明|
|---|---|
|**无序性**|不保留插入顺序，按字典序自动排序|
|**唯一性**|同一元素在集合中只能出现一次|
|**高效查找**|可快速判断元素是否存在|
|**集合运算**|天然支持并集、交集等操作（基于排序）|
|**空键自动清理**|当删除集合的最后一个元素时，LMDB 会自动删除该键（因为 LMDB 不允许空键）|

### 3.2 LMDB 空键行为说明

由于 `set` 基于 LMDB 的重复键（`MDB_DUPSORT`）实现，需要注意：

```
// 添加元素
set[ptr_set] → "apple"
set[ptr_set] → "banana"

// 删除元素
删除 "apple" → set[ptr_set] 仍保留 "banana"
删除 "banana" → set[ptr_set] 被 LMDB 自动删除（键不存在）
```

> ⚠️ **重要**：当集合为空时，该指针在 `set` 中将不存在。这是 LMDB 的正常行为，不应视为错误。上层应用需要自行处理空集合的情况（如通过其他元数据记录集合的存在性）。

### 3.3 对象的空成员列表行为

**对象同样受此影响**，因为对象的成员名列表也存储在 `set` 中：

```
// 对象有两个成员
set[ptr_obj] → "age"
set[ptr_obj] → "name"
main[ptr_obj + "age"] → 30
main[ptr_obj + "name"] → "Alice"

// 删除成员
删除 "age" → set[ptr_obj] 仍保留 "name"
删除 "name" → set[ptr_obj] 被 LMDB 自动删除（键不存在）
```

> ⚠️ **重要**：当对象的所有成员都被删除后，该对象指针在 `set` 中将不存在。**对象的注册信息本身也依赖于 `set`**：
> - 对象指针**不再有效**——因为实体的存在性由 `set` 定义
> - 遍历 `set` 时无法发现该对象
> - 上层应用需要通过其他方式追踪空对象的存在性（如维护一个对象注册表）

### 3.3.1 删除成员的安全保证

**关键设计保证**：成员的删除必须遵循"先删值，再删注册"的顺序：

```
// 删除成员 "name" 的正确流程
1. 删除 main[ptr_obj + "name"] → 先删除值
2. 删除 set[ptr_obj] → "name"  → 再删除注册
```

由于成员注册（在 `set` 中）和成员值（在 `main` 中）是**绑定删除**的：
- ✅ 可以单独删除成员值（保留注册，表示"有成员但值为空"）
- ❌ 不能单独删除成员注册（会导致幽灵成员）
- ✅ **当 `set` 中成员列表为空时，`main` 中必然没有对应的成员值**

> 💡 **这意味着**：删除对象的最后一个成员时，虽然 `set` 中的键被 LMDB 自动删除，但**不会产生幽灵成员问题**——因为成员值的删除总是先于成员注册的删除。

### 3.4 设计影响与应对策略

| 场景 | 问题 | 建议方案 |
|------|------|----------|
| 空集合 | 指针在 `set` 中消失，实体不再有效 | **避免创建空集合**；或在应用层维护一个"空集合标记"（在 `main` 中存储一个特殊值） |
| 空对象 | 指针在 `set` 中消失，实体不再有效 | **避免删除对象的最后一个成员**；或保留一个占位成员（如 `_empty`） |
| 遍历所有实体 | 无法发现空实体 | 维护独立的全局索引/注册表（仅记录指针，不存储结构） |

> 💡 **核心原则**：**实体的存在性由 `set` 定义**。无论是对象还是集合，只有在 `set` 中有注册（即存在以其指针为 Key 的条目）时，该实体才被视为有效存在。
>
> ⚠️ **重要限制**：外部索引**无法替代 `set` 的作用**。如果 `set` 中没有定义，即使外部索引记录了指针，也无法解析实体的结构（对象的成员列表、集合的元素列表）。因此：
> - **推荐方案**：避免创建会变为空的实体
> - **替代方案**：在删除最后一个元素前，在 `main` 中存储一个占位值作为"空实体标记"

### 3.5 如果需要有序列表怎么办？

若应用需要保留顺序（如时间线、排行榜），必须在 Value 中显式编码顺序信息，例如：

```c
// 格式：[4B 小端序下标][原始值]
set[ptr_set] → "\x00\x00\x00\x00banana"
set[ptr_set] → "\x00\x00\x00\x01apple"
```

读取后按前 4 字节排序即可还原原始顺序。

> 💡 **设计理念**：内核只提供基础的无序集合，顺序是应用层语义。这种设计保持了内核的简洁性，同时为上层提供了充分的灵活性。

## 4. 嵌套结构：靠指针链接

复杂结构通过**指针引用**实现：

```js
{
  user: { name: "Bob" },   // user 是一个对象
  tags: ["a", "b"]         // tags 是一个集合
}
```

- `main[parent_ptr + "user"] → child_obj_ptr`（17B）
- `main[parent_ptr + "tags"] → tags_set_ptr`（17B）
- `child_obj_ptr` 和 `tags_set_ptr` 分别指向 `set`/`main` 中定义的子实体

→ 整个树状结构被**彻底扁平化**，但可通过指针完整重建。

### 嵌套存储示例

```
父对象指针: ptr_parent (01...)

set[ptr_parent] → "user"
set[ptr_parent] → "tags"

main[ptr_parent + "user"] → ptr_child (01...)
main[ptr_parent + "tags"] → ptr_tags (02...)

--- 子对象 user ---
set[ptr_child] → "name"
main[ptr_child + "name"] → "Bob"

--- 子集合 tags ---
set[ptr_tags] → "a"
set[ptr_tags] → "b"
```

## 5. 一致性规则

为保障数据正确性，LMJCore 强制以下规则：

| 规则                                | 说明                         |
| --------------------------------- | -------------------------- |
| **成员必须先在 `set` 中声明**              | 不能直接向 `main` 写入未注册的成员      |
| **`main` 中存在但 `set` 中缺失 → 幽灵成员**  | 属于严重数据损坏                   |
| **`set` 中存在但 `main` 中缺失 → 合法缺失值** | 表示该成员尚未赋值                  |
| **Key 长度 ≤ 511 字节**               | 成员名最长 493 字节（17B 指针 + 成员名） |
| **集合元素唯一性**                      | 同一集合中不能重复添加相同元素            |

### 5.1 幽灵成员检测

```c
// 审计对象，检测幽灵成员
int lmjcore_audit_object(lmjcore_txn *txn, const uint8_t *obj_ptr, 
                          audit_callback_fn callback);
```

幽灵成员表明数据损坏，应立即：
1. 停止写入操作
2. 调用修复函数
3. 记录审计日志

## 6. 为什么这样设计？

|优势|说明|
|---|---|
|**极致读性能**|对象成员可并行点查 `main`，无需解析嵌套结构|
|**局部更新**|修改单个成员只需更新一个 `main` 格子|
|**灵活扩展**|上层可自由决定是否编码顺序、类型等元信息|
|**继承 LMDB 优势**|ACID、零拷贝、内存映射、崩溃安全|
|**集合运算高效**|基于排序的集合天然支持交集、并集等操作|

## 7. 使用场景对比

### 7.1 对象 vs 集合

|维度|对象|集合|
|---|---|---|
|**指针标识**|`0x01`|`0x02`|
|**结构**|键值对（成员名 → 值）|元素列表（无序）|
|**存储位置**|成员名在 `set`，值在 `main`|元素在 `set`|
|**访问方式**|通过成员名点查|遍历或判断存在性|
|**典型场景**|配置、用户信息、文档|标签、权限组、唯一值列表|

### 7.2 选择指南

```c
// 需要键值对 → 用对象
{
  "name": "Alice",
  "age": 30,
  "email": "alice@example.com"
}

// 只需要元素列表 → 用集合
["admin", "editor", "viewer"]

// 需要有序列表 → 在集合中编码顺序
[
  {"seq": 0, "value": "first"},
  {"seq": 1, "value": "second"}
]
```

## 总结：LMJCore 的存储心智模型

- **`set` 是“集合区”**：给定一个指针，你能拿到它所有的关联项（成员名或集合元素），但这些项是**无序集合**，自动去重和排序。
- **`main` 是“主存储区”**：每个格子由“指针 + 名称”唯一标识，值独立存储、强隔离。
- **指针是“胶水”**：将分散的片段重新组合成完整的逻辑实体。
- **顺序是应用语义**：内核不提供，但也不阻止——你可以在 Value 中编码任何排序逻辑。

> 这种设计让 LMJCore 在 **灵活性**（无模式、任意嵌套）与 **性能**（扁平、直接寻址）之间取得了极佳平衡。
